# Collection & Detail Six-Model Alignment Design

**Date:** 2026-05-03
**Project:** T.R.Y.P frontend showroom
**Scope:** Align homepage, collection, and detail pages around the same 6-car catalog and replace old static imagery with renders captured from the 3D models.

## 1. Goal

The site currently uses different car lists across homepage, collection, and detail views. The collection/detail flow shows 7 cars that do not match the homepage lineup, and some detail pages display a 3D model that does not correspond to the selected car.

This change standardizes the experience so the full site uses exactly these 6 models:

1. Murciélago
2. 3.0 CSL Hommage
3. 488 Pista Spider
4. Roma
5. Centenario Roadster
6. Centenario LP-770

The collection page and detail page must both reflect only those 6 models. The detail page `360° View` must load the correct model for the selected car, and the framing must feel closer to the homepage showroom reference instead of leaving the car small in the scene.

## 2. Source Of Truth

`assets/js/data.js` becomes the single source of truth for the collection and detail flows.

The shared catalog will contain exactly 6 entries in the same visual order the user expects from the homepage. Each entry must define:

- Stable `id` used in URLs such as `detail.html?car=...`
- `brand`, `name`, and optional `subtitle`
- Display specs: `hp`, `engine`, `acceleration`, `topSpeed`, `price`, `year`
- `model3d` path pointing to the correct `.glb`
- `modelConfig` for detail page `360° View`
- `images.hero`
- `images.gallery` with the standardized render set

Recommended ids:

- `murcielago`
- `bmw-3-0-csl-hommage`
- `ferrari-488-pista-spider`
- `ferrari-roma`
- `centenario-roadster`
- `centenario-lp770`

The existing 7-car multi-brand data set in `data.js` is out of scope after this alignment and should no longer drive collection/detail rendering.

## 3. Canonical Model Mapping

Each detail page must use the correct model and no fallback cross-mapping between unrelated cars.

Canonical mapping:

| Car | Model file |
| --- | --- |
| Murciélago | `assets/model/lambo_lp670.glb` |
| 3.0 CSL Hommage | `assets/model/2015_bmw_3.0_csl_hommage_concept.glb` |
| 488 Pista Spider | `assets/model/2019_ferrari_488_pista_spider.glb` |
| Roma | `assets/model/2020_ferrari_roma.glb` |
| Centenario Roadster | `assets/model/lamborghini_centenario_roadster_sdc.glb` |
| Centenario LP-770 | `assets/model/lamborghini_centenario_lp-770_interior_sdc.glb` |

If a model file fails to load, the page may show a graceful fallback message, but the data mapping itself must still remain one-to-one and explicit.

## 4. Rendered Image Strategy

Existing static car images are no longer authoritative for the aligned 6-car experience. New images will be generated by capturing still renders directly from each 3D model.

Every model will receive the same standardized 4-angle image set:

- `hero`: front 3/4 angle, used for collection cards and detail hero
- `front`: lower front-facing angle
- `rear`: rear 3/4 angle
- `top`: high 3/4 or semi-top angle

Recommended storage pattern:

`assets/img/cars/<car-id>/hero.webp`
`assets/img/cars/<car-id>/front.webp`
`assets/img/cars/<car-id>/rear.webp`
`assets/img/cars/<car-id>/top.webp`

This image set will be used as follows:

- Collection card: `hero`
- Detail hero: `hero`
- Detail gallery: `hero`, `front`, `rear`, `top`
- Compare cards: `hero`

The old mixed image library can remain on disk temporarily, but collection/detail rendering should stop depending on it for these 6 entries.

## 5. Camera And Framing Rules For 360 View

The detail page `360° View` should match the visual intent of the homepage showroom reference more closely:

- Car appears larger in frame than it does now
- Camera sits closer to the vehicle
- Composition emphasizes the whole body shape without excessive empty background
- Framing remains elegant and centered rather than aggressively cropped

Each car gets its own `modelConfig`, but all configs should follow a shared framing rule:

- `cameraOrbit`: tuned per model to present a front 3/4 showroom angle
- `cameraTarget`: adjusted so the car sits visually balanced in the container
- `fieldOfView`: narrowed or widened only as needed to keep the car prominent
- Optional `exposure`: normalized if some models render too dark

The default experience remains rotation-focused:

- Keep `camera-controls`
- Keep `disable-pan`
- Keep zoom restricted unless implementation requires a slight relaxation for usability
- Preserve the dark luxury background and uncluttered stage

## 6. Collection Page Behavior

`catalog.html` will render exactly 6 cards from the shared catalog data.

Requirements:

- Remove the current 7-car output
- Preserve the existing visual language where reasonable
- Use the new rendered `hero` image per model
- Link each card to `detail.html?car=<id>`
- Ensure only the 6 approved models appear in booking dropdowns tied to this flow

No extra cars, hidden legacy entries, or mismatched labels should remain in the rendered collection UI.

## 7. Detail Page Behavior

`detail.html` must become a strict projection of the selected entry from the shared 6-car data set.

Requirements:

- Sidebar title/specs must match the selected car
- Hero image must use the rendered `hero` image of that same car
- Gallery must use the 4 render captures from that same car
- `360° View` must load the matching `.glb` for that same car
- Compare dropdown must only list the other 5 approved models
- Invalid or missing query ids may fall back to the first valid entry, but never to legacy removed cars

This removes the current inconsistency where the page title, static image, and 3D model can represent different vehicles.

## 8. Asset Capture Workflow

To keep the visuals consistent, still images should be captured programmatically from the models rather than manually screenshotting arbitrary browser states.

Expected workflow:

1. Load each model into a controlled viewer scene
2. Apply the preset camera for each standardized angle
3. Capture the output to image files
4. Save the results into the new per-car image directory
5. Wire those generated files back into `data.js`

Benefits:

- Consistent framing across all 6 cars
- Fast iteration if the user wants closer or wider crops later
- Reproducible asset generation instead of one-off manual screenshots

## 9. Files Expected To Change

Primary implementation targets:

- `assets/js/data.js`
- `assets/js/catalog.js`
- `assets/js/detail.js`
- `assets/html/catalog.html`
- `assets/html/detail.html`
- `assets/css/detail.css`
- `assets/img/cars/...` newly generated render assets
- Optional capture utility under `tools/` or a similar project script location

Homepage code may be referenced for visual matching, but this change does not require redesigning the homepage itself unless shared ids need a small alignment adjustment.

## 10. Success Criteria

The work is complete when all of the following are true:

1. Homepage, collection, and detail all refer to the same 6 approved models.
2. Collection shows exactly 6 cars and no legacy extras.
3. Every collection card opens the correct detail page for that same model.
4. Each detail page uses one internally consistent set of content: title, specs, gallery images, and 3D model all describe the same car.
5. Each detail page has a 4-image gallery generated from the corresponding model.
6. The default `360° View` framing is noticeably closer and visually aligned with the homepage showroom reference.

## 11. Out Of Scope

The following are not part of this design:

- Adding new cars beyond the approved 6 models
- Backend or CMS changes
- Checkout or real booking integration
- Homepage structural redesign unrelated to six-model alignment
- Reworking the brand taxonomy beyond what is required to label the 6 approved entries
